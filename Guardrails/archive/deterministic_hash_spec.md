# Deterministic Hash Spec

## Goal

PRANA hashes must be stable across replays.

Required property:

`Same semantic input -> always same SHA-256 hash`

This document defines the PRANA hashing boundary used by runtime ingestion and replay orchestration.

## Hashing Pipeline

1. Normalize the input recursively.
2. Remove nondeterministic fields.
3. Canonicalize the normalized structure into strict JSON.
4. Compute SHA-256 over the canonical UTF-8 JSON bytes.

## Canonical JSON Rules

- Objects are serialized with keys sorted lexicographically.
- JSON separators are fixed to `,` and `:`.
- ASCII-safe JSON encoding is used.
- Nested object ordering is normalized recursively.
- List ordering is preserved as-is because list order is treated as semantic.

## Nondeterministic Field Removal

These fields are excluded before hashing:

- exact keys:
  - `event_id`
  - `validation_id`
- any key containing:
  - `timestamp`
  - `uuid`
- UUID-valued identifiers where the key is `id` or ends with `_id`

Examples of stripped fields:

- `event_timestamp`
- `replay_timestamp`
- `received_at`
- `freshness_timestamp`
- `event_id`
- `validation_id`
- `trace_uuid`
- `submission_id` only when the value itself is a UUID

## Float Normalization

PRANA treats numerically equivalent float forms as identical.

These values normalize to the same hash representation:

- `1`
- `1.0`
- `1.000`

Non-integral floats are normalized to a trailing-zero-free decimal string.

Examples:

- `1.2000 -> "1.2"`
- `0.500 -> "0.5"`
- `100.0 -> 100`

Non-finite floats are not allowed:

- `NaN`
- `Infinity`
- `-Infinity`

## Boundary Intent

Included in the hash:

- semantic payload content
- sequence numbers
- event type content
- run identifiers
- source-relevant fields that describe the event meaning

Excluded from the hash:

- runtime-generated timestamps
- replay-generated timestamps
- UUID-style identifiers generated by the system
- validation IDs

## Replay Implication

Replay comparison must use the same normalization rules as ingestion.

That means:

- ingestion hash and replay hash are computed from the same normalized representation
- differing timestamps or UUIDs do not cause false mismatches
- reordered nested objects do not cause false mismatches
- float formatting differences do not cause false mismatches

## Practical Outcome

If two PRANA events mean the same thing, they hash the same even when:

- timestamps differ
- UUIDs differ
- nested object key order differs
- floats are represented differently

This is the determinism boundary for PRANA.
